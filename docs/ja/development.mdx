このガイドでは、F5 XC API エンリッチメントパイプラインの開発ワークフローについて説明します。

## クイックスタート

### 初回セットアップ

```bash
# リポジトリをクローン
git clone https://github.com/f5-sales-demo/api-specs-enriched.git
cd api-specs-enriched

# 仮想環境を作成して依存関係をインストール
make install

# pre-commit フックをインストール
make pre-commit-install

# 仕様をダウンロードしてパイプラインを実行
make build

# ドキュメントをローカルでプレビュー
make serve
```

### 日常の開発

```bash
# クイックリビルド（キャッシュ済み仕様を使用）
make rebuild

# すべての品質チェックを実行
make pre-commit-run

# ドキュメントをプレビュー
make serve
```

## ワークフローパターン

### 1. ディスカバリーワークフロー

**前提条件**:
- F5 XC 環境への VPN 接続
- 有効な API 認証情報

```bash
# 認証情報を設定
export F5XC_API_URL="https://your-tenant.console.ves.volterra.io/api"
export F5XC_API_TOKEN="your-api-token"

# ディスカバリーを実行
make discover

# 結果を確認
jq '.statistics' specs/discovered/session.json

# CI/CD 用にコミット
make push-discovery
```

**ディスカバリーで取得される情報**:
- 実際の必須/任意フィールド
- ライブレスポンスからの列挙値
- デフォルト値
- パターンバリデーション
- レスポンス例

### 2. リリースワークフロー

リリースは自動化されています:

1. 毎日のスケジュール（UTC 午前 6 時）またはメインへのプッシュでワークフローがトリガーされる
2. ETag チェックで F5 仕様が変更されたか判定する
3. パイプラインが仕様を処理してエンリッチする
4. バージョンは git タグから計算される: `git describe --tags --abbrev=0`
5. 直接コミット + タグが作成される（バージョンバンプ PR なし）
6. 変更履歴付きの GitHub リリースが作成される
7. ドキュメントが GitHub Pages にデプロイされる

**バージョンバンプルール**:

| 条件 | バンプ種別 | 例 |
|-----------|-----------|---------|
| コミットに `[major]` を含む | メジャー | 1.0.0 → 2.0.0 |
| コミットに `BREAKING CHANGE` を含む | メジャー | 1.0.0 → 2.0.0 |
| 新しいドメイン仕様が追加された | マイナー | 1.0.0 → 1.1.0 |
| その他の変更 | パッチ | 1.0.0 → 1.0.1 |

### 3. 開発ワークフロー

```bash
# フィーチャーブランチを作成
git checkout -b feature/my-change

# 設定またはスクリプトに変更を加える

# ローカルでテスト
make pipeline
make lint

# コミット（pre-commit フックが自動的に実行される）
git add .
git commit -m "feat: add new enrichment rule"

# プッシュして PR を作成
git push -u origin feature/my-change
gh pr create
```

## ディスカバリー詳細解説

### ディスカバリーとは？

ディスカバリーはライブ F5 XC API を探索して以下を検出します:

- **未文書化の制約**: OpenAPI にマークされていない必須フィールド
- **実際の列挙値**: 本番環境で確認された実際の値
- **デフォルト動作**: フィールドが省略された場合にサーバーが適用する値
- **レスポンスパターン**: 実際のデータ形状

### ディスカバリー設定

```yaml
# config/discovery.yaml
discovery:
  exploration:
    namespaces:
      - "system"
      - "shared"
    methods:
      - "GET"
      - "OPTIONS"
    max_endpoints_per_run: 500

  schema_inference:
    sample_size: 3
    detect_patterns: true
    detect_constraints: true
```

### ディスカバリー拡張

ディスカバリーは `x-discovered-*` 拡張を仕様に追加します:

```json
{
  "properties": {
    "name": {
      "type": "string",
      "x-discovered-required": true,
      "x-discovered-pattern": "^[a-z][a-z0-9-]*$",
      "x-discovered-examples": ["my-app", "prod-lb"]
    }
  }
}
```

### 制約分析レポート

```bash
make constraint-report
# 出力: reports/constraint-analysis.md
```

## OpenAPI 拡張

エンリッチメントパイプラインはベンダー拡張を使用して、バリデーションとデフォルト値のメタデータを埋め込みます。

### 拡張カテゴリ

| 拡張 | 意味 | 含意 |
|-----------|---------|-------------|
| `x-f5xc-server-default` | F5 XC API サーバーがフィールド省略時にこの値を適用する | フィールドは任意であり、省略すると文書化されたデフォルト動作が生成される |
| `x-f5xc-recommended-value` | F5 XC Web コンソールが新規リソース作成時にこの値をプリポピュレートする | フィールドにサーバーデフォルトはないが、この値が典型的な設定を表す |
| `x-f5xc-recommended-oneof-variant` | F5 XC コンソールがこの OneOf バリアントを事前選択する | 複数の相互排他オプションが存在する場合の典型的な選択を識別する |
| `x-f5xc-conflicts-with` | このフィールドと同時に使用できない他のプロパティを列挙する | プロパティは OneOf グループの一部であり、競合するプロパティの1つのみ指定できる |

### 必須フィールド拡張

| 拡張 | ソース | 目的 |
|-----------|--------|---------|
| `x-ves-required: "true"` | F5 XC 元仕様 | フィールドにゼロ以外/空以外の値が必要 |
| `x-f5xc-required-for` | エンリッチメントパイプライン | コンテキスト固有の必須ステータス |

**x-f5xc-required-for コンテキスト**:
- `create`: リソース作成時に必須
- `update`: リソース更新時に必須
- `minimum_config`: 最小限の実行可能な設定に必須

### デフォルト値と OneOf 拡張

| 拡張 | 目的 |
|-----------|---------|
| `x-f5xc-server-default: true` | サーバーが適用するデフォルトをマーク |
| `x-f5xc-recommended-value` | F5 XC コンソールがプリポピュレートする値 |
| `x-f5xc-recommended-oneof-variant` | 推奨 OneOf バリアント |
| `x-f5xc-conflicts-with` | 他の OneOf プロパティとの相互排他性 |

#### x-f5xc-server-default

**型**: `boolean`

`true` の場合、付随する `default` 値が F5 XC API サーバーによって強制されることを示します。この拡張を持つフィールドは API リクエストから安全に省略できます — サーバーが自動的にデフォルトを適用します。

```yaml
use_http2:
  type: boolean
  default: false
  x-f5xc-server-default: true
```

#### x-f5xc-recommended-value

**型**: `any`（フィールド型に対応）

F5 XC Web コンソールがプリポピュレートされたデフォルトとして使用する値を指定します。この値はサーバーによって強制されませんが、コンソールを通じて作成された新規リソースの典型的な初期設定を表します。

```yaml
timeout:
  type: integer
  x-f5xc-recommended-value: 3
```

#### x-f5xc-recommended-oneof-variant

**型**: `object`（グループ名からバリアント名へのマップ）

相互排他フィールドグループを持つスキーマにおいて、どのバリアントがデフォルトまたは最も一般的な選択かを識別します。キーは OneOf グループ名、値は推奨バリアントのフィールド名です。

```yaml
healthcheckCreateSpecType:
  type: object
  x-f5xc-recommended-oneof-variant:
    health_check: "http_health_check"
```

#### x-f5xc-conflicts-with

**型**: 文字列の `array`

**追加**: v3.2.0（Issue #494）

OneOf グループメンバー間の相互排他関係を宣言します。`x-ves-oneof-field-*` 拡張から自動導出されます。これにより、ダウンストリームツール（Terraform、CLI、MCP）がランタイムではなくスキーマレベルで競合を検証できます。

```yaml
host_header:
  type: string
  x-f5xc-conflicts-with: ["use_origin_server_name"]

use_origin_server_name:
  type: object
  x-f5xc-conflicts-with: ["host_header"]
```

**ユースケース**:
- Terraform プロバイダーがバリデーションルールを生成できる
- CLI ツールが競合するフィールドの組み合わせについて警告できる
- AI アシスタントが正しい OneOf 設定を生成できる
- IDE 拡張機能が競合を考慮したオートコンプリートを提供できる

**ソース**: パイプラインエンリッチメント中に F5 ネイティブの `x-ves-oneof-field-*` 拡張から自動導出。

### バリデーションルールの含意

エンリッチャーは `x-ves-validation-rules` を検査して必須ステータスを推定します:

| ルール | 含意 |
|------|-------------|
| `ves.io.schema.rules.message.required: "true"` | フィールドは必須 |
| `ves.io.schema.rules.uint32.gte: N` | N >= 1 かつサーバーデフォルトがない場合、フィールドは必須 |
| `ves.io.schema.rules.repeated.min_items: N` | N >= 1 の場合、配列には少なくとも N 個のアイテムが必要 |
| `ves.io.schema.rules.string.min_bytes: N` | N >= 1 の場合、文字列には少なくとも N バイトが必要 |

### サーバー適用デフォルトを持つリソース

一部のリソースはサーバーがデフォルトを適用するため、空のスペックを受け入れます:

| リソース | サーバー適用デフォルト |
|----------|------------------------|
| `app_firewall` | `monitoring: \{\}`, `default_detection_settings: \{\}` |
| `rate_limiter` | `limits: []`, `user_identification: []` |
| `api_definition` | `swagger_specs: []`, デフォルト `api_groups` |
| `healthcheck` | `jitter: 0`, `jitter_percent: 0`, ネストされた http_health_check デフォルト |

これらのリソースでは、`x-ves-required` が `true` の場合でも `x-f5xc-required-for.create` が `false` になる場合があります。

### デフォルト値パターン

| パターン | 型 | 例 |
|---------|------|---------|
| `\{\}` | 空オブジェクト（選択肢の選択） | `monitoring: \{\}` |
| `[]` | 空配列 | `expected_status_codes: []` |
| `0` | 数値 | `jitter: 0` |
| `""` | 文字列 | `expected_response: ""` |
| `false` | 真偽値 | `use_http2: false` |

### 検出済みデフォルトの追加

1. API 経由で最小スペックを使用して F5 XC でリソースを作成する
2. リソースを読み返してサーバーが適用した値を確認する
3. `config/discovered_defaults.yaml` にデフォルトを記録する
4. パイプラインを実行する: `make pipeline`
5. 次のコマンドで確認する: `jq '.components.schemas | to_entries[] | select(.key | contains("resource_name"))'`

## トラブルシューティング

### Pre-commit の処理が遅すぎる

パイプラインは一貫性を確保するためにコミットごとに実行されます。作業中のコミットの場合:

```bash
git commit --no-verify -m "WIP: work in progress"
# 最終コミット前に実行: make pre-commit-run
```

### ディスカバリーが失敗する

```bash
# VPN を確認
ping your-tenant.console.ves.volterra.io

# 認証情報を確認
echo $F5XC_API_TOKEN | head -c 10

# API URL フォーマットを確認
echo $F5XC_API_URL
# フォーマット: https://tenant.console.ves.volterra.io/api
```

### 生成された仕様のリントエラー

```bash
# リントレポートを確認
cat reports/lint-report.json | jq '.errors'
```

出力ファイルではなく、エンリッチメント/正規化設定で問題を修正してください。

### バージョンシステム

バージョンは git タグから導出されます（例: `v2.0.38`）。これにより、並行する PR でのマージコンフリクトを引き起こしていた競合状態が解消されます。

```python
# バージョンは git タグから動的に計算される
from scripts.utils.version_calculator import get_version_from_tags
version = get_version_from_tags()  # "2.0.38" を返す
```

### クローン後に仕様が見つからない

生成された仕様は gitignored です:

```bash
make build  # すべてをダウンロードして生成する
```

### 大きなファイルがブロックされる

`.pre-commit-config.yaml` に除外設定を追加します:

```yaml
- id: check-added-large-files
  exclude: ^path/to/large/file\.json$
```
