このガイドでは、F5 XC API エンリッチメントパイプラインの開発ワークフローについて説明します。

## クイックスタート

### 初回セットアップ

```bash
# リポジトリをクローン
git clone https://github.com/f5xc-salesdemos/api-specs-enriched.git
cd api-specs-enriched

# 仮想環境を作成し、依存関係をインストール
make install

# pre-commit フックをインストール
make pre-commit-install

# スペックをダウンロードしてパイプラインを実行
make build

# ドキュメントをローカルでプレビュー
make serve
```

### 日常の開発

```bash
# クイックリビルド（キャッシュされたスペックを使用）
make rebuild

# すべての品質チェックを実行
make pre-commit-run

# ドキュメントをプレビュー
make serve
```

## ワークフローパターン

### 1. ディスカバリワークフロー

**前提条件**:
- F5 XC 環境への VPN 接続
- 有効な API 認証情報

```bash
# 認証情報を設定
export F5XC_API_URL="https://your-tenant.console.ves.volterra.io/api"
export F5XC_API_TOKEN="your-api-token"

# ディスカバリを実行
make discover

# 結果を確認
jq '.statistics' specs/discovered/session.json

# CI/CD 用にコミット
make push-discovery
```

**ディスカバリで取得される情報**:
- 実際の必須/任意フィールド
- ライブレスポンスからの列挙値
- デフォルト値
- パターンバリデーション
- レスポンスの例

### 2. リリースワークフロー

リリースは自動化されています：

1. 日次スケジュール（UTC 午前6時）または main へのプッシュでワークフローがトリガーされます
2. ETag チェックにより F5 スペックの変更を判定します
3. パイプラインがスペックを処理・エンリッチします
4. バージョンは git タグから計算されます: `git describe --tags --abbrev=0`
5. 直接コミット + タグが作成されます（バージョンバンプ PR なし）
6. 変更ログ付きの GitHub Release が作成されます
7. ドキュメントが GitHub Pages にデプロイされます

**バージョンバンプルール**:

| 条件 | バンプタイプ | 例 |
|-----------|-----------|---------|
| コミットに `[major]` を含む | メジャー | 1.0.0 → 2.0.0 |
| コミットに `BREAKING CHANGE` を含む | メジャー | 1.0.0 → 2.0.0 |
| 新しいドメインスペックが追加された | マイナー | 1.0.0 → 1.1.0 |
| その他の変更 | パッチ | 1.0.0 → 1.0.1 |

### 3. 開発ワークフロー

```bash
# フィーチャーブランチを作成
git checkout -b feature/my-change

# 設定やスクリプトを変更

# ローカルでテスト
make pipeline
make lint

# コミット（pre-commit フックが自動的に実行されます）
git add .
git commit -m "feat: add new enrichment rule"

# プッシュして PR を作成
git push -u origin feature/my-change
gh pr create
```

## ディスカバリの詳細

### ディスカバリとは？

ディスカバリは、ライブの F5 XC API を調査して以下を検出します：

- **ドキュメント化されていない制約**: OpenAPI でマークされていない必須フィールド
- **実際の列挙値**: 本番環境で確認された実際の値
- **デフォルトの動作**: フィールドが省略された場合にサーバーが適用する値
- **レスポンスパターン**: 実際のデータ構造

### ディスカバリ設定

```yaml
# config/discovery.yaml
discovery:
  exploration:
    namespaces:
      - "system"
      - "shared"
    methods:
      - "GET"
      - "OPTIONS"
    max_endpoints_per_run: 500

  schema_inference:
    sample_size: 3
    detect_patterns: true
    detect_constraints: true
```

### ディスカバリ拡張

ディスカバリはスペックに `x-discovered-*` 拡張を追加します：

```json
{
  "properties": {
    "name": {
      "type": "string",
      "x-discovered-required": true,
      "x-discovered-pattern": "^[a-z][a-z0-9-]*$",
      "x-discovered-examples": ["my-app", "prod-lb"]
    }
  }
}
```

### 制約分析レポート

```bash
make constraint-report
# 出力: reports/constraint-analysis.md
```

## OpenAPI 拡張

エンリッチメントパイプラインは、バリデーションおよびデフォルト値のメタデータを埋め込むためにベンダー拡張を使用します。

### 拡張カテゴリ

| 拡張 | 意味 | 影響 |
|-----------|---------|-------------|
| `x-f5xc-server-default` | フィールドが省略された場合に F5 XC API サーバーがこの値を適用する | フィールドは任意であり、省略するとドキュメント化されたデフォルトの動作が適用される |
| `x-f5xc-recommended-value` | F5 XC ウェブコンソールが新しいリソースに対してこの値をプリセットする | フィールドにサーバーデフォルトはないが、この値が一般的な設定を表す |
| `x-f5xc-recommended-oneof-variant` | F5 XC コンソールがこの OneOf バリアントをデフォルト選択する | 相互排他的な複数のオプションがある場合の一般的な選択肢を特定する |
| `x-f5xc-conflicts-with` | このフィールドと同時に使用できない他のプロパティを一覧表示する | プロパティは OneOf グループの一部であり、競合するプロパティのうち1つのみ指定可能 |

### 必須フィールド拡張

| 拡張 | ソース | 用途 |
|-----------|--------|---------|
| `x-ves-required: "true"` | F5 XC オリジナルスペック | フィールドにゼロ以外/空以外の値が必要 |
| `x-f5xc-required-for` | エンリッチメントパイプライン | コンテキスト固有の必須ステータス |

**x-f5xc-required-for のコンテキスト**:
- `create`: リソース作成時に必須
- `update`: リソース更新時に必須
- `minimum_config`: 最小限の実用的な設定に必須

### デフォルト値と OneOf 拡張

| 拡張 | 用途 |
|-----------|---------|
| `x-f5xc-server-default: true` | サーバー適用のデフォルトをマーク |
| `x-f5xc-recommended-value` | F5 XC コンソールのプリセット値 |
| `x-f5xc-recommended-oneof-variant` | 推奨される OneOf バリアント |
| `x-f5xc-conflicts-with` | 他の OneOf プロパティとの相互排他性 |

#### x-f5xc-server-default

**型**: `boolean`

`true` の場合、付随する `default` 値が F5 XC API サーバーによって適用されることを示します。この拡張を持つフィールドは API リクエストから安全に省略できます。サーバーが自動的にデフォルトを適用します。

```yaml
use_http2:
  type: boolean
  default: false
  x-f5xc-server-default: true
```

#### x-f5xc-recommended-value

**型**: `any`（フィールドの型に一致）

F5 XC ウェブコンソールがプリセットのデフォルトとして使用する値を指定します。この値はサーバーによって強制されませんが、コンソール経由で作成される新しいリソースの一般的な初期設定を表します。

```yaml
timeout:
  type: integer
  x-f5xc-recommended-value: 3
```

#### x-f5xc-recommended-oneof-variant

**型**: `object`（グループ名からバリアント名へのマップ）

相互排他的なフィールドグループを持つスキーマにおいて、デフォルトまたは最も一般的な選択肢となるバリアントを特定します。キーは OneOf グループ名、値は推奨されるバリアントのフィールド名です。

```yaml
healthcheckCreateSpecType:
  type: object
  x-f5xc-recommended-oneof-variant:
    health_check: "http_health_check"
```

#### x-f5xc-conflicts-with

**型**: 文字列の `array`

**追加**: v3.2.0（Issue #494）

OneOf グループメンバー間の相互排他性の関係を宣言します。`x-ves-oneof-field-*` 拡張から自動的に導出されます。これにより、下流ツール（Terraform、CLI、MCP）がランタイムではなくスキーマレベルで競合を検証できるようになります。

```yaml
host_header:
  type: string
  x-f5xc-conflicts-with: ["use_origin_server_name"]

use_origin_server_name:
  type: object
  x-f5xc-conflicts-with: ["host_header"]
```

**ユースケース**:
- Terraform プロバイダーがバリデーションルールを生成可能
- CLI ツールが競合するフィールドの組み合わせについて警告可能
- AI アシスタントが正しい OneOf 設定を生成可能
- IDE 拡張が競合を考慮したオートコンプリートを提供可能

**ソース**: パイプラインのエンリッチメント処理中に F5 ネイティブの `x-ves-oneof-field-*` 拡張から自動的に導出されます。

### バリデーションルールの影響

エンリッチャーは `x-ves-validation-rules` を検査して必須ステータスを推論します：

| ルール | 影響 |
|------|-------------|
| `ves.io.schema.rules.message.required: "true"` | フィールドは必須 |
| `ves.io.schema.rules.uint32.gte: N` | N >= 1 かつサーバーデフォルトがない場合、フィールドは必須 |
| `ves.io.schema.rules.repeated.min_items: N` | N >= 1 の場合、配列には少なくとも N 個のアイテムが必要 |
| `ves.io.schema.rules.string.min_bytes: N` | N >= 1 の場合、文字列には少なくとも N バイトが必要 |

### サーバー適用デフォルトを持つリソース

一部のリソースは、サーバーがデフォルトを適用するため空のスペックを受け入れます：

| リソース | サーバー適用デフォルト |
|----------|------------------------|
| `app_firewall` | `monitoring: \{\}`, `default_detection_settings: \{\}` |
| `rate_limiter` | `limits: []`, `user_identification: []` |
| `api_definition` | `swagger_specs: []`, デフォルトの `api_groups` |
| `healthcheck` | `jitter: 0`, `jitter_percent: 0`, ネストされた http_health_check のデフォルト |

これらのリソースでは、`x-ves-required` が `true` であっても `x-f5xc-required-for.create` が `false` になる場合があります。

### デフォルト値パターン

| パターン | 型 | 例 |
|---------|------|---------|
| `\{\}` | 空オブジェクト（選択指定） | `monitoring: \{\}` |
| `[]` | 空配列 | `expected_status_codes: []` |
| `0` | 数値 | `jitter: 0` |
| `""` | 文字列 | `expected_response: ""` |
| `false` | ブール | `use_http2: false` |

### 検出されたデフォルトの追加

1. API 経由で最小限のスペックで F5 XC にリソースを作成します
2. リソースを読み戻してサーバー適用の値を確認します
3. `config/discovered_defaults.yaml` にデフォルトを記録します
4. パイプラインを実行: `make pipeline`
5. 検証: `jq '.components.schemas | to_entries[] | select(.key | contains("resource_name"))'`

## トラブルシューティング

### Pre-commit に時間がかかりすぎる

一貫性を確保するため、パイプラインはすべてのコミットで実行されます。作業中のコミットの場合：

```bash
git commit --no-verify -m "WIP: work in progress"
# 最終コミット前に実行: make pre-commit-run
```

### ディスカバリが失敗する

```bash
# VPN を確認
ping your-tenant.console.ves.volterra.io

# 認証情報を確認
echo $F5XC_API_TOKEN | head -c 10

# API URL の形式を確認
echo $F5XC_API_URL
# 形式: https://tenant.console.ves.volterra.io/api
```

### 生成されたスペックのリントエラー

```bash
# リントレポートを確認
cat reports/lint-report.json | jq '.errors'
```

問題はエンリッチメント/正規化の設定で修正してください。出力ファイルを直接修正しないでください。

### バージョンシステム

バージョンは git タグ（例: `v2.0.38`）から導出され、同時実行の PR でマージコンフリクトを引き起こすレースコンディションを排除します。

```python
# バージョンは git タグから動的に計算されます
from scripts.utils.version_calculator import get_version_from_tags
version = get_version_from_tags()  # Returns "2.0.38"
```

### クローン後にスペックがない

生成されたスペックは gitignore 対象です：

```bash
make build  # すべてをダウンロードして生成します
```

### 大きなファイルがブロックされる

`.pre-commit-config.yaml` に除外設定を追加してください：

```yaml
- id: check-added-large-files
  exclude: ^path/to/large/file\.json$
```
