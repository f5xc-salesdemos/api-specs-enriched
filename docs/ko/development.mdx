이 가이드는 F5 XC API 보강 파이프라인의 개발 워크플로우를 설명합니다.

## 빠른 시작

### 최초 설정

```bash
# Clone the repository
git clone https://github.com/f5xc-salesdemos/api-specs-enriched.git
cd api-specs-enriched

# Create virtual environment and install dependencies
make install

# Install pre-commit hooks
make pre-commit-install

# Download specs and run pipeline
make build

# Preview documentation locally
make serve
```

### 일상적인 개발

```bash
# Quick rebuild (uses cached specs)
make rebuild

# Run all quality checks
make pre-commit-run

# Preview documentation
make serve
```

## 아키텍처 개요

```text

             + Grammar    + Types    + Master     rules
```

## 워크플로우 패턴

### 1. 디스커버리 워크플로우

**사전 요구 사항**:
- F5 XC 환경에 대한 VPN 연결
- 유효한 API 자격 증명

```bash
# Set credentials
export F5XC_API_URL="https://your-tenant.console.ves.volterra.io/api"
export F5XC_API_TOKEN="your-api-token"

# Run discovery
make discover

# View results
jq '.statistics' specs/discovered/session.json

# Commit for CI/CD
make push-discovery
```

**디스커버리 수집 항목**:
- 실제 필수/선택 필드
- 실시간 응답의 열거형(Enum) 값
- 기본값
- 패턴 유효성 검사
- 응답 예시

### 2. 릴리스 워크플로우

릴리스는 자동화되어 있습니다:

1. 일일 스케줄(UTC 오전 6시) 또는 main 브랜치에 푸시 시 워크플로우 실행
2. ETag 확인을 통해 F5 사양 변경 여부 결정
3. 파이프라인이 사양을 처리하고 보강
4. git 태그에서 버전 계산: `git describe --tags --abbrev=0`
5. 직접 커밋 + 태그 생성 (버전 업 PR 없음)
6. 변경 이력을 포함한 GitHub 릴리스 생성
7. GitHub Pages에 문서 배포

**버전 업 규칙**:

| 조건 | 업 유형 | 예시 |
|-----------|-----------|---------|
| 커밋에 `[major]` 포함 | Major | 1.0.0 → 2.0.0 |
| 커밋에 `BREAKING CHANGE` 포함 | Major | 1.0.0 → 2.0.0 |
| 새 도메인 사양 추가 | Minor | 1.0.0 → 1.1.0 |
| 기타 변경 사항 | Patch | 1.0.0 → 1.0.1 |

### 3. 개발 워크플로우

```bash
# Create feature branch
git checkout -b feature/my-change

# Make changes to config or scripts

# Test locally
make pipeline
make lint

# Commit (pre-commit hooks run automatically)
git add .
git commit -m "feat: add new enrichment rule"

# Push and create PR
git push -u origin feature/my-change
gh pr create
```

## 디스커버리 심층 분석

### 디스커버리란?

디스커버리는 실제 F5 XC API를 탐색하여 다음을 찾습니다:

- **문서화되지 않은 제약 조건**: OpenAPI에 표시되지 않은 필수 필드
- **실제 열거형 값**: 프로덕션에서 확인된 실제 값
- **기본 동작**: 필드가 생략될 때 서버에서 적용하는 값
- **응답 패턴**: 실제 데이터 형태

### 디스커버리 구성

```yaml
# config/discovery.yaml
discovery:
  exploration:
    namespaces:
      - "system"
      - "shared"
    methods:
      - "GET"
      - "OPTIONS"
    max_endpoints_per_run: 500

  schema_inference:
    sample_size: 3
    detect_patterns: true
    detect_constraints: true
```

### 디스커버리 확장

디스커버리는 사양에 `x-discovered-*` 확장을 추가합니다:

```json
{
  "properties": {
    "name": {
      "type": "string",
      "x-discovered-required": true,
      "x-discovered-pattern": "^[a-z][a-z0-9-]*$",
      "x-discovered-examples": ["my-app", "prod-lb"]
    }
  }
}
```

```bash
make constraint-report
```


## OpenAPI 확장

보강 파이프라인은 벤더 확장을 사용하여 유효성 검사 및 기본값 메타데이터를 임베드합니다.

### 확장 카테고리

| 확장 | 의미 | 함의 |
|-----------|---------|-------------|
| `x-f5xc-server-default` | F5 XC API 서버가 필드를 생략했을 때 이 값을 적용 | 필드는 선택 사항이며, 생략 시 문서화된 기본 동작을 생성 |
| `x-f5xc-recommended-value` | F5 XC 웹 콘솔이 새 리소스에 대해 이 값을 미리 채움 | 서버 기본값은 없지만 이 값이 일반적인 구성을 나타냄 |
| `x-f5xc-recommended-oneof-variant` | F5 XC 콘솔이 이 OneOf 변형을 미리 선택 | 상호 배타적인 여러 옵션이 있을 때 일반적인 선택을 식별 |
| `x-f5xc-conflicts-with` | 이 필드와 함께 사용할 수 없는 다른 속성 목록 | 속성이 OneOf 그룹의 일부이며, 충돌하는 속성 중 하나만 지정 가능 |

### 필수 필드 확장

| 확장 | 출처 | 목적 |
|-----------|--------|---------|
| `x-ves-required: "true"` | F5 XC 원본 사양 | 필드에 비어있지 않은 값 필요 |
| `x-f5xc-required-for` | 보강 파이프라인 | 컨텍스트별 필수 상태 |

**x-f5xc-required-for 컨텍스트**:
- `create`: 리소스 생성 시 필수
- `update`: 리소스 업데이트 시 필수
- `minimum_config`: 최소 실행 가능 구성에 필수

### 기본값 및 OneOf 확장

| 확장 | 목적 |
|-----------|---------|
| `x-f5xc-server-default: true` | 서버에서 적용하는 기본값 표시 |
| `x-f5xc-recommended-value` | F5 XC 콘솔 미리 채워진 값 |
| `x-f5xc-recommended-oneof-variant` | 권장 OneOf 변형 |
| `x-f5xc-conflicts-with` | 다른 OneOf 속성과의 상호 배타성 |

#### x-f5xc-server-default

**타입**: `boolean`

`true`일 때, 함께 제공되는 `default` 값이 F5 XC API 서버에 의해 강제 적용됨을 나타냅니다. 이 확장이 있는 필드는 API 요청에서 안전하게 생략할 수 있으며, 서버가 기본값을 자동으로 적용합니다.

```yaml
use_http2:
  type: boolean
  default: false
  x-f5xc-server-default: true
```

#### x-f5xc-recommended-value

**타입**: `any` (필드 타입과 일치)

F5 XC 웹 콘솔이 미리 채워진 기본값으로 사용하는 값을 지정합니다. 이 값은 서버에서 강제 적용되지 않지만 콘솔을 통해 생성된 새 리소스의 일반적인 시작 구성을 나타냅니다.

```yaml
timeout:
  type: integer
  x-f5xc-recommended-value: 3
```

#### x-f5xc-recommended-oneof-variant

**타입**: `object` (그룹 이름에서 변형 이름으로의 맵)

상호 배타적인 필드 그룹이 있는 스키마에서, 어떤 변형이 기본값 또는 가장 일반적인 선택인지 식별합니다. 키는 OneOf 그룹 이름이고 값은 권장 변형 필드 이름입니다.

```yaml
healthcheckCreateSpecType:
  type: object
  x-f5xc-recommended-oneof-variant:
    health_check: "http_health_check"
```

#### x-f5xc-conflicts-with

**타입**: 문자열의 `array`

**추가됨**: v3.2.0 (Issue #494)

OneOf 그룹 구성원 간의 상호 배타성 관계를 선언합니다. `x-ves-oneof-field-*` 확장에서 자동으로 파생됩니다. 이를 통해 다운스트림 도구(Terraform, CLI, MCP)가 런타임이 아닌 스키마 수준에서 충돌을 검증할 수 있습니다.

```yaml
host_header:
  type: string
  x-f5xc-conflicts-with: ["use_origin_server_name"]

use_origin_server_name:
  type: object
  x-f5xc-conflicts-with: ["host_header"]
```

**활용 사례**:
- Terraform 프로바이더가 유효성 검사 규칙 생성 가능
- CLI 도구가 충돌하는 필드 조합에 대해 경고 가능
- AI 어시스턴트가 올바른 OneOf 구성 생성 가능
- IDE 확장이 충돌 인식 자동 완성 제공 가능

**출처**: 파이프라인 보강 과정에서 F5 네이티브 `x-ves-oneof-field-*` 확장으로부터 자동 파생됩니다.

### 유효성 검사 규칙 함의

보강기는 `x-ves-validation-rules`를 검사하여 필수 상태를 추론합니다:

| 규칙 | 함의 |
|------|-------------|
| `ves.io.schema.rules.message.required: "true"` | 필드가 필수 |
| `ves.io.schema.rules.uint32.gte: N` | N >= 1이고 서버 기본값이 없으면 필드가 필수 |
| `ves.io.schema.rules.repeated.min_items: N` | N >= 1이면 배열에 최소 N개 항목 필요 |
| `ves.io.schema.rules.string.min_bytes: N` | N >= 1이면 문자열에 최소 N바이트 필요 |

### 서버 적용 기본값이 있는 리소스

일부 리소스는 서버가 기본값을 적용하므로 빈 사양을 허용합니다:

| 리소스 | 서버 적용 기본값 |
|----------|------------------------|
| `app_firewall` | `monitoring: \{\}`, `default_detection_settings: \{\}` |
| `rate_limiter` | `limits: []`, `user_identification: []` |
| `api_definition` | `swagger_specs: []`, 기본 `api_groups` |
| `healthcheck` | `jitter: 0`, `jitter_percent: 0`, 중첩된 http_health_check 기본값 |

이러한 리소스의 경우, `x-ves-required`가 `true`이더라도 `x-f5xc-required-for.create`가 `false`일 수 있습니다.

### 기본값 패턴

| 패턴 | 타입 | 예시 |
|---------|------|---------|
| `\{\}` | 빈 객체 (선택 사항 선택) | `monitoring: \{\}` |
| `[]` | 빈 배열 | `expected_status_codes: []` |
| `0` | 숫자 | `jitter: 0` |
| `""` | 문자열 | `expected_response: ""` |
| `false` | 불리언 | `use_http2: false` |

### 디스커버된 기본값 추가

1. API를 통해 최소 사양으로 F5 XC에 리소스 생성
2. 리소스를 다시 읽어 서버 적용 값 확인
3. `config/discovered_defaults.yaml`에 기본값 문서화
4. 파이프라인 실행: `make pipeline`
5. 다음으로 확인: `jq '.components.schemas | to_entries[] | select(.key | contains("resource_name"))'`

## 문제 해결

### 사전 커밋이 너무 오래 걸림

파이프라인은 일관성을 보장하기 위해 모든 커밋에서 실행됩니다. 작업 중인 커밋의 경우:

```bash
git commit --no-verify -m "WIP: work in progress"
# Run before final commit: make pre-commit-run
```

### 디스커버리 실패

```bash
# Check VPN
ping your-tenant.console.ves.volterra.io

# Check credentials
echo $F5XC_API_TOKEN | head -c 10

# Check API URL format
echo $F5XC_API_URL
# Format: https://tenant.console.ves.volterra.io/api
```

### 생성된 사양의 린트 오류

```bash
# Check lint report
cat reports/lint-report.json | jq '.errors'
```

출력 파일이 아닌 보강/정규화 구성에서 문제를 수정하세요.

### 버전 시스템

버전은 git 태그(예: `v2.0.38`)에서 파생되므로, 동시 PR에서 병합 충돌을 유발했던 경쟁 조건이 제거됩니다.

```python
# Version is calculated dynamically from git tags
from scripts.utils.version_calculator import get_version_from_tags
version = get_version_from_tags()  # Returns "2.0.38"
```

### 클론 후 사양이 없음

생성된 사양은 gitignore 처리됩니다:

```bash
make build  # Downloads and generates everything
```

### 대용량 파일 차단

`.pre-commit-config.yaml`에 제외 항목 추가:

```yaml
- id: check-added-large-files
  exclude: ^path/to/large/file\.json$
```
